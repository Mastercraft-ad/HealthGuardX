## Functional workflow & rules (exact behavior)

* **On first connect:**

  * If wallet not registered, present registration modal. On register action call contract registerUser/ registerPatient flow (as applicable). After registration, user remains in restricted dashboard until KYC submitted and approved.
  * If wallet registers as admin address exactly `0x3c17f3F...`, present Admin dashboard immediately.
* **KYC process:**

  * KYC docs must be encrypted client-side, uploaded to decentralized storage, and the CID submitted on-chain or via a secure off-chain tracker with hash on-chain. Admin sees KYC as a CID and can approve/reject on-chain.
* **Uploading records:**

  * Client-side generate symmetric key per file, encrypt file, upload encrypted file to decentralized storage, compute CID and file integrity hash, and store CID pointer on-chain using the contract’s update function.
  * For access, patient will create wrappedKey blobs encrypted to grantee public keys and addWrappedKey on-chain. Client must implement ECIES/ECDH wrapping and store wrapped blobs as bytes referenced by the contract.
* **QR behavior:**

  * QR contains the UID or secure vault pointer only — no PII or unencrypted data. Scanning a QR triggers a read of the contract for emergency summary or full CID pointers based on the scanner’s wallet role. Public scans show only emergency summary and a clear notice of limited data.
* **Emergency grant escalation:**

  * Emergency pre-auth: patient may pre-authorize addresses that auto-grant temporary wrappedKey expiry. If not pre-authorized, emergency request emits event and admin or guardians may grant emergency access; any emergency grant must be time-limited and logged.
* **Claims & Insurance:**

  * Hospital/Doctor submits claim with proofCid; insurer reviews, approves/rejects on-chain; on approval the insurer (or admin in demo mode) triggers releasePayout which simulates transfer and writes payout event.
* **Audit & transparency:**

  * All critical events (register, kyc submitted/approved, wrapped key added/removed, emergency request/grant, claim submit/approve/reject, payout) must be read as events and displayed in real time throughout the UI.
